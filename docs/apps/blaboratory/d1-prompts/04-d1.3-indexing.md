```
Implement Phase D1.3 of the Blaboratory vector-retrieval system. Branch
blaboratory-part2. Goal: embed-and-store all not-yet-indexed events/chat/
utterances rows in one batched, idempotent pass — keeping embed calls OUT of the
synchronous write_phase.

Read first:
- docs/apps/blaboratory/d1-vector-build-plan.md — "Phase D1.3" + "How we use it"
  (corpus + indexing-trigger).
- Phase D1.1 committed code (vector_index.VectorIndex.add, vec_rowmap) and D1.2b
  (embed_texts).
- app/apps/blaboratory/context_pipeline.gather_memories — reuse its line-rendering
  shape so indexed text == the text a resident would read.

Build (additive):
- app/apps/blaboratory/memory_index.py:
  - render_indexable(row, kind) -> str: the same short line shape as
    gather_memories (e.g. "[tick N] you <action>: <summary>", chat as
    "[tick N] chat from <author>: <body>", utterances similarly).
  - index_pending(*, limit=...): find rows in events/chat/utterances NOT present
    in vec_rowmap (LEFT JOIN by (kind, ref_id)); embed in batches via
    embed_texts(is_query=False); VectorIndex.add with resident_id/kind/ref_id/tick;
    record vec_rowmap. Degrade to a logged no-op if not VEC_AVAILABLE or the embed
    server is unreachable (never raise into the tick).

Done when tests/apps/test_memory_index.py passes (patched embed returning fixed
vectors):
- seed events + chat → index_pending indexes each exactly once (vec_rowmap +
  vec_memories counts match);
- a second index_pending call is a no-op;
- an embed outage (embed_texts raises) leaves rows un-indexed WITHOUT raising.
Full existing suite stays green.

Env: .venv/bin/python, .venv/bin/pytest (asyncio_mode=auto), py_compile.
Don't commit until I've reviewed.
```
