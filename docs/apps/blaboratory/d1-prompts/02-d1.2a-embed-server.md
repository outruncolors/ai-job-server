# D1.2a — App-managed embed `llama-server` + control routes

Phase **D1.2a** of [`../d1-vector-build-plan.md`](../d1-vector-build-plan.md).
**Prev:** [01 — D1.1 schema](01-d1.1-schema-index.md) · **Next:** [03 — D1.2b embed client](03-d1.2b-embed-client.md) · [Sequence index](README.md)

This is the piece that gets **committed and deployed to the secondary** so it runs the embedder.
Independent of D1.1 — can be built in parallel. Buildable/testable locally with subprocess+httpx
mocks; real verification needs the LLM node with the bge-small GGUF in `/opt/ai-stack/models/`.

```
Implement Phase D1.2a of the Blaboratory vector-retrieval system. Branch
blaboratory-part2. Goal: a second, always-on llama-server OWNED BY THE APP on
llm-capable nodes, serving /v1/embeddings on port 8081 — the sibling of the chat
LlamaCppManager. The same app artifact runs on both primary and the gpu (llm)
node, so this is controlled through the app, then deployed.

Read first:
- docs/apps/blaboratory/d1-vector-build-plan.md — "Phase D1.2a" + the Topology
  section.
- app/llamacpp/manager.py (entire) — adopt/start/_health_ok/_spawn/_read_loop/
  _terminate/_wait_ready, the _stable_hash swap lock, os.killpg cleanup, the
  500-line ring buffer. Mirror this.
- app/llamacpp/config.py and app/llamacpp/router.py — config + route shapes.
- app/main.py lifespan — where the chat LlamaCppManager starts/adopts; note the
  capability gating ("llm" in get_local_capabilities()).

Build (additive; capability-gated so non-llm nodes never spawn it):
1. app/llamacpp/embed_manager.py: LlamaCppEmbedManager. Fixed embed preset →
   argv: --model <embed_model_path> --embeddings --pooling cls --host <host>
   --port <embed_port> --ctx-size 512 -ngl 99. adopt-if-already-running on the
   embed port; 180s readiness via /health; ring-buffer logs; killpg teardown.
   (Confirm --pooling cls is correct for bge against the pinned LLAMA_CPP_TAG
   b9204 — wrong pooling silently degrades similarity.)
2. Config: add embed_port / embed_model_path / embed_pooling to llamacpp/config.py
   (or a dedicated config/llamacpp_embed.json with the same get/save pattern).
3. Routes /v1/llamacpp-embed/{status,start,stop,restart,logs} mirroring
   llamacpp/router.py.
4. app/main.py lifespan: start/adopt the embed manager on llm nodes, stop on
   shutdown (beside the chat manager).

Done when tests/test_embed_manager.py passes (subprocess + httpx mocked like the
existing llama.cpp manager tests): start spawns the embed argv on the embed port;
adopt picks up an already-running server; readiness timeout → clear error +
cleared state; routes return status. Full existing suite stays green.
Real-node manual check (note in PR, not automated):
  curl localhost:8081/v1/embeddings -d '{"input":"hi","model":"bge-small"}'
  → a 384-length embedding.

Env: .venv/bin/python, .venv/bin/pytest (asyncio_mode=auto), py_compile.
Don't commit until I've reviewed.
```

> **Deploy note:** after this merges, the embed model file must exist on the secondary
> (`/opt/ai-stack/models/bge-small-en-v1.5-f16.gguf`) and the deployed `config` must carry
> `embed_port`/`embed_model_path`. See D1.5 for the deploy checklist.
